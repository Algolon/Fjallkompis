# Focused design-review direction — Today stage block

## Status

**Owner-approved implementation direction; validation pending.**

This note records an iterative decision made during Design Review #1. It is not a final release judgement and does not complete the full v0.18 pre-field review.

## Review context

The Today screen is intended to function as the operational hub of the hiking day. The current `Day X of 7` action block provides access to the Map through **View Route**, but it does not provide direct access to the detailed guide for the currently selected stage.

A simple additional button was considered insufficient because the review discussion surfaced two connected needs:

1. Today should provide a quick, glanceable understanding of the current stage without duplicating the full Stage Guide.
2. The block must remain vertically bounded so the Today page does not become scroll-dependent.

The proposed solution must also preserve Fjällkompis' battery-conscious, offline-first character. It must not depend on continuous GPS or dynamic live context.

## Accepted design direction

Redesign the `Day X of 7` block so it contains:

1. the existing day and stage-title context;
2. up to four static, non-interactive stage-highlight chips;
3. a **Stage Guide** action that opens the currently selected stage on the Stages page;
4. a **View Route** action that opens the Map focused on the currently selected stage while preserving existing in-session map context behaviour.

### Stage-highlight chips

- Each chip contains an icon and a concise one-line label.
- The chips represent need-to-know stage characteristics, such as steep climbing, rocky terrain, boat availability, exposure, limited water, bridge crossings, or the destination type.
- No separate heading such as `Today's Highlights` or `Stage Highlights` is shown.
- Chips are metadata, not controls.
- Chips are static stage metadata and do not depend on GPS, live tracking, time of day, or network access.
- Display no more than four chips.
- If more than four characteristics apply, use a predefined priority order and show only the four most important.
- Display fewer than four when fewer are genuinely useful; do not fill space artificially.
- The layout must be tested with three and four chips at the smallest supported phone viewport.
- Four chips are acceptable only when the block remains visually calm, labels remain readable, and the Today page preserves its no-unnecessary-scroll behaviour.
- Prefer shorter labels or a maximum of three chips over reducing text legibility or creating overflow.

### Actions

- **Stage Guide** opens the full guide for the currently selected stage.
- **View Route** opens the Map focused on that stage.
- The implementation must not reset a user's remembered in-session Map view except when the user explicitly invokes a stage-specific action from Today.
- Both actions should be clearly discoverable without turning Today into a general-purpose navigation menu.

## Fixed responsibility and anti-growth guardrail

The `Day X of 7` block has a fixed responsibility:

> Present today's stage, today's essential stage characteristics, and the two primary follow-up actions.

New information should only be added when it replaces existing content or demonstrably reduces cognitive load. The block must not gradually become a general-purpose dashboard. Weather, detailed elevation, live progress, destination facilities, explanatory paragraphs, or other useful information should remain in their authoritative screens unless a future Design Authority decision explicitly changes this boundary.

## Design Authority assessment

### First-time user

The Stage Guide becomes discoverable without requiring prior knowledge of the Stages tab. The chips help a first-time user understand the character of the day at a glance.

### Experienced hiker

The block surfaces exceptional or operationally relevant stage characteristics rather than repeating the full guide. It supports a short morning check without pretending to replace detailed planning or primary navigation.

### Interaction and information architecture

The design creates progressive depth:

`Today summary → Stage Guide detail → Map context`

The two actions have distinct responsibilities, and the non-interactive chips avoid ambiguous tap behaviour.

### Visual interface

Omitting a highlight heading preserves vertical space and prevents the block from fragmenting into multiple labelled subsections. The chip group must remain compact and integrated into the existing Nordic card language.

### Accessibility and outdoor usability

Labels must remain legible and must not be compressed merely to fit four chips. Icons must supplement rather than replace text. Colour alone must not communicate meaning.

### Systems and software integrity

Highlights should be derived from structured stage metadata so they remain deterministic, testable, offline and transferable. No background GPS or dynamic state is introduced.

### Creator intent

The direction strengthens Today as immediate decision support without duplicating the Stage Guide or turning Today into another feature catalogue. It preserves the no-unnecessary-scroll principle and Fjällkompis' battery-conscious product identity.

## Validation requirements

Before merging an implementation:

- compare the current block with the proposed block at all supported phone widths;
- render stages with zero, one, two, three and four highlights;
- verify the longest allowed labels do not create horizontal overflow;
- verify four chips do not force avoidable Today-page scrolling at the smallest supported height;
- verify Stage Guide opens the currently selected stage;
- verify View Route opens that stage on Map;
- verify the remembered Map state still behaves as previously designed when navigating away and back normally;
- verify chips are semantically non-interactive and accessible;
- run the existing test, typecheck and build gates.

## Evidence classification

- Current missing Stage Guide access: **Observed**.
- Risk of duplicating the full Stage Guide on Today: **Derived**.
- Risk of vertical overflow and unwanted scrolling: **Observed design constraint / derived implementation risk**.
- Battery impact of dynamic GPS-dependent highlights: **Derived**, consistent with the existing battery-conscious product boundary.
- Final choice between a three-chip practical maximum and allowing four: **Implementation experiment requiring viewport evidence**.

## Decision

**Proceed with implementation as a bounded design experiment.**

The implementation may support up to four highlights, but four is not a target to fill. Evidence from responsive validation should determine whether a specific stage uses three or four. Any solution that causes overflow, reduced legibility, visual crowding, or Today-page scrolling must fall back to fewer or shorter highlights.
